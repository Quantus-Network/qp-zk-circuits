\title{Wormhole Addresses: Recursive Zero-Knowledge Private Settlement for Post-Quantum Blockchains}
\author{Christopher Smith \and Ethan Cemer}
\date{March 2026}

\begin{document}
\maketitle

\begin{abstract}
Post-quantum signature schemes impose large public keys and signatures, increasing on-chain costs while still leaking transaction structure. We present \emph{Quantus Wormhole}: users deposit to addresses derived from secrets, then prove in zero knowledge that (i) the deposit exists in authenticated state, (ii) the block-header commitment is valid, (iii) the nullifier is fresh, and (iv) exits satisfy fee constraints. Recursive aggregation amortizes verification across many withdrawals. The first aggregation layer pads batches with dummy proofs and shuffles locally to enlarge the explanation class for balance-linking adversaries. We formalize the leaf relation, analyze layer-0 privacy, prove deposit binding and one-time withdrawal, and report proving times for batch sizes 2--49.
\end{abstract}

\section{Introduction}

Shor's algorithm breaks elliptic-curve signatures \citep{shor}. Post-quantum replacements like ML-DSA have much larger keys and signatures, increasing bandwidth, state growth, and verification cost \citep{mldsa}. Simply swapping signature schemes sacrifices throughput and does nothing for privacy.

We take a different approach: instead of authorizing spends directly on-chain, users deposit to addresses derived from secrets and later prove in zero knowledge that (i) the deposit exists, (ii) they know the secret, (iii) it hasn't been spent, and (iv) exits satisfy fee rules. Recursive aggregation then compresses many withdrawals into fewer on-chain verifications.

This design builds on EIP-7503's wormhole concept \citep{eip7503}, adapted to the Quantus stack with storage-proof-based implementation \citep{quantusrepo}.

\paragraph{Contributions.}
\begin{enumerate}[leftmargin=1.5em]
    \item Formalize the \emph{wormhole-address} spending model: deposits flow to deterministic unspendable addresses derived from a secret; withdrawals require proving knowledge of that secret plus authenticated state inclusion.
    \item Specify the leaf relation: public outputs, private witnesses, and sub-circuit constraints for nullifier derivation, unspendable-account derivation, storage-proof verification, fee-constrained dual exits, and block-header commitment.
    \item Describe the recursive aggregation architecture. The first aggregation layer batches leaf proofs sharing the same \texttt{block\_hash}, pads with dummy leaves, shuffles client-locally, and wraps with a circuit forwarding nullifiers and grouped exits.
    \item Provide combinatorial privacy analysis of layer-0, showing how dummy padding, hidden batch order, and grouped exits enlarge the explanation class for balance-linking adversaries.
    \item Explain why the public commitment covers the full block header (not just the state root), letting the runtime validate that the witnessed state root belongs to canonical history.
    \item Report measured build, prove, and verify times using \texttt{qp-plonky2} with Poseidon2 recursion, for batch sizes 2--49 (layer-0) and fan-ins 2--32 (layer-1).
    \item State security claims for deposit binding, wrapper soundness, one-time withdrawal, spend-path exclusivity, and value conservation, separating proof-level from runtime-level assumptions.
\end{enumerate}

\section{Background and Design Goals}

\subsection{From post-quantum signatures to proof-based settlement}

Post-quantum signatures are expensive to publish and verify. Rather than publishing a large signature per spend, users can prove knowledge of a short secret checked against authenticated state. The chain verifies a proof and updates a nullifier set.

\subsection{Relationship to wormholes and privacy pools}

EIP-7503 proposes \emph{zero-knowledge wormholes}: deposit to secret-derived addresses, then prove knowledge of the secret to withdraw \citep{eip7503}. Quantus differs in three ways: (1) post-quantum-friendly proving, (2) implemented against authenticated state (not just a proposal), and (3) recursive aggregation with dummy padding.

Related systems include privacy pools and shielded ledgers like Zcash and Tornado Cash \citep{eip7503,privacypools,zcash,tornadocash}. Unlike fully shielded ledgers, Quantus proves against native chain state without a separate note commitment tree.

\subsection{Why transparent recursion}

We use \texttt{qp-plonky2}, combining PLONK arithmetization with FRI commitments for recursive verification without trusted setup \citep{plonky2,fri,qpplonky2}. Poseidon2 provides efficient in-circuit hashing over prime fields \citep{poseidon2}.

\subsection{Design goals}

The protocol aims to satisfy four simultaneous goals:
\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{Scalable authorization.} On-chain work should scale with proof verification and forwarded public outputs, not with raw post-quantum signature material per transaction.
    \item \textbf{Authenticated settlement.} Every successful withdrawal must be bound to a real deposit recorded in authenticated chain state.
    \item \textbf{Replay prevention.} A deposit should not authorize multiple withdrawals; nullifiers must uniquely bind proofs to deposits.
    \item \textbf{Meaningful privacy.} The chain should reveal only the public information needed for settlement: exits, amounts, shared fee parameters, nullifiers, and a block commitment. Funding accounts, secrets, and raw storage proofs should remain private.
\end{enumerate}

\section{System, Adversary, and Trust Model}
\label{sec:model}

\subsection{Roles}

The system comprises the following roles.
\begin{itemize}[leftmargin=1.5em]
    \item \textbf{User / wallet.} Generates a secret, derives an wormhole address, funds it, collects the witness data for withdrawal, and may optionally perform first-layer aggregation locally.
    \item \textbf{Proof source or indexer.} Supplies the user with the relevant transfer record, block-header data, and storage proof needed to construct a witness. This role is not trusted for correctness because incorrect data causes proof failure.
    \item \textbf{First-layer aggregator.} A privacy-sensitive local actor that fills a batch with real and dummy leaf proofs, shuffles them, and produces the first recursive wrapper.
    \item \textbf{Higher-layer aggregator.} A potentially delegated actor that composes already aggregated proofs and forwards their public outputs.
    \item \textbf{Runtime / verifier.} Checks block consistency, fee parameters, nullifier freshness, and proof validity before applying settlement.
\end{itemize}

\subsection{State objects}

Let \(\mathcal{A}\) be the set of 32-byte account identifiers and let \(\sigma\) denote chain state. A withdrawal witness references the following objects:
\begin{itemize}[leftmargin=1.5em]
    \item an asset identifier \(a\),
    \item an input amount \(v_{\mathrm{in}}\), quantized to the circuit's 0.01-unit resolution,
    \item a transfer counter \(c\),
    \item a funding account \(\alpha_{\mathrm{fund}}\),
    \item an unspendable deposit account \(\alpha_{\mathrm{wh}}\),
    \item a processed storage proof authenticating the transfer record under a state root \(\mathsf{rt}\), and
    \item structured block-header fields \((\mathsf{parent\_hash}, b, \mathsf{rt}, \mathsf{extrinsics\_root}, \mathsf{digest})\).
\end{itemize}

The runtime additionally maintains a nullifier set \(\mathcal{N}\) of all accepted nullifiers.

\subsection{Adversarial capabilities}

We assume a probabilistic polynomial-time adversary that can observe the full chain, submit arbitrary deposits, inspect public proof outputs, perform timing analysis, and attempt to replay or malform proof submissions. The adversary does \emph{not} break the collision resistance of the hash function, the knowledge soundness of the proof system, or the chain's block-history availability assumptions.

\subsection{Trust assumptions}

The protocol relies on the following assumptions.
\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{Hash security.} The Poseidon2 instantiation used by the system remains collision resistant and preimage resistant for the relevant security level \citep{poseidon2}.
    \item \textbf{Proof soundness.} The configured Plonky2 recursion stack remains knowledge sound under its standard assumptions \citep{plonky2,fri}.
    \item \textbf{Authenticated history.} The runtime can check that the public block commitment and block number correspond to a valid block in chain history.
    \item \textbf{Witness availability.} Honest users can obtain the storage proof and block-header data needed to build a witness for their deposit.
\end{enumerate}

\section{Protocol Overview}
\label{sec:overview}

The protocol consists of five stages.

\paragraph{1. Secret generation and deposit.}
The user samples a secret \(s\) and derives an wormhole address
\[
\WA(s) = H(H(\mathsf{``wormhole''} \parallel s)).
\]
The user then funds this address through an ordinary transfer on Quantus. The deposit is accompanied by a chain record that can later be authenticated inside the circuit.

\paragraph{2. Witness acquisition.}
To withdraw, the user gathers the transfer record, a processed storage proof, the state root to which that proof authenticates, and the structured block-header fields needed to recompute the public block commitment. These data form the private witness except for the public block fields that will be exposed by the proof.

A subtlety is that the protocol does \emph{not} expose the state root directly as its public chain anchor. An earlier design considered doing exactly that, but a bare public state root is insufficient for runtime validation: seeing only a root does not tell the verifier whether that root belongs to any valid canonical block. The implemented design therefore witnesses the structured header privately and exposes a public Poseidon2 commitment to that full header together with the public block number. The runtime validates the committed header against chain history, while the circuit binds the privately used state root in the storage proof to that same committed header.

\paragraph{3. Leaf proof generation.}
The user chooses up to two exit outputs, computes a nullifier
\[
\Null(s,c) = H(H(\mathsf{``\string~nullif\string~''} \parallel s \parallel c)),
\]
and produces a leaf proof. The proof reveals the nullifier, the exits, the output amounts, the asset identifier, the fee basis points, the public block commitment, and the block number, while keeping the secret, funding account, storage proof, and transfer-specific raw details private.

\paragraph{4. First-layer aggregation.}
A privacy-sensitive local aggregator batches leaf proofs that share the same public \texttt{block\_hash}. If the batch is not full, the aggregator pads it with dummy leaves, shuffles the order, recursively merges the proofs, and then applies a wrapper circuit that enforces consistency and forwards grouped exits and nullifiers.

\paragraph{5. Settlement.}
The runtime verifies the aggregate proof, checks that the public \texttt{block\_hash} matches the referenced block number, verifies that the fee parameter matches runtime configuration, rejects reused nullifiers, and settles the exits.

\begin{algorithm}[t]
\caption{High-level withdrawal pipeline}
\label{alg:pipeline}
\begin{algorithmic}[1]
\Procedure{WithdrawAndSettle}{$s,c,\alpha_1,\alpha_2,v_1,v_2$}
    \State derive \(\alpha_{\mathrm{wh}} \gets \WA(s)\)
    \State fetch transfer record, block-header fields, and storage proof for counter \(c\)
    \State compute \(n \gets \Null(s,c)\)
    \State form public inputs \(x = (a,v_1,v_2,f,n,\alpha_1,\alpha_2,h_B,b)\)
    \State form witness \(w\) from secret, transfer record, and authenticated state data
    \State produce leaf proof \(\pi \gets \mathsf{Prove}(x,w)\)
    \State optionally aggregate \(\pi\) with same-block leaf proofs
    \State submit proof and public inputs for runtime verification
\EndProcedure
\end{algorithmic}
\end{algorithm}

\subsection{Leaf relation}

Let \(x = (a, v_1, v_2, f, n, \alpha_1, \alpha_2, h_B, b)\) denote the public input tuple and let \(w\) denote the witness described above. The leaf relation \(\mathcal{R}_{\mathrm{leaf}}\) holds when there exists a witness \(w\) such that:
\begin{enumerate}[label=(C\arabic*),leftmargin=1.9em]
    \item \textbf{Nullifier correctness.} \(n = \Null(s,c)\).
    \item \textbf{Entangled-address correctness.} \(\alpha_{\mathrm{wh}} = \WA(s)\).
    \item \textbf{Authenticated inclusion.} The processed storage proof verifies that a transfer record keyed by the relevant funding account, wormhole account, and counter exists under \(\mathsf{rt}\).
    \item \textbf{Header binding.} The structured block-header fields hash to the public commitment \(h_B\), and the public block number equals the header's block number.
    \item \textbf{Fee constraint.} The disclosed exits satisfy
    \[
    (v_1 + v_2) \cdot 10000 \leq v_{\mathrm{in}} \cdot (10000 - f).
    \]
    \item \textbf{Range constraints.} The public quantities lie within the ranges expected by the runtime interface.
\end{enumerate}

\begin{definition}[Leaf relation]
The relation \(\mathcal{R}_{\mathrm{leaf}}\) is the set of all \((x,w)\) satisfying conditions (C1)--(C6).
\end{definition}

\subsection{Runtime interface}

The proof system does not, by itself, settle a withdrawal. The runtime must also check the following: (i) the referenced block exists in chain history, (ii) the public fee parameter matches the active runtime fee configuration, (iii) the public nullifier has not appeared before, and (iv) any asset-specific accounting rules are satisfied. These external checks are part of the overall security argument.

\section{Recursive Aggregation Architecture}
\label{sec:aggregation}

\subsection{First-layer policy}

The first aggregation layer is privacy-sensitive. A valid batch must satisfy:
\begin{enumerate}[leftmargin=1.9em]
    \item Every real leaf proof in the batch shares the same public \texttt{block\_hash}.
    \item Every real leaf proof shares the same \texttt{asset\_id} and \texttt{volume\_fee\_bps}.
    \item If fewer than \(N\) real leaves are available, the batch is padded with dummy leaves.
    \item The leaf order is shuffled before wrapping, subject to implementation-specific handling that preserves at least one real leaf in a stable position for correctness.
    \item Duplicate exit accounts are grouped so that identical destinations accumulate a summed amount in one slot while duplicate slots are zeroed.
    \item All nullifiers are forwarded, with dummy nullifiers replaced by fresh random values.
\end{enumerate}

Because the padding and shuffle are part of the privacy mechanism, layer-0 should be executed locally by the wallet whenever possible; a delegated layer-0 service would observe the client's raw child proofs before they are hidden inside the wrapper and would therefore collapse much of the ambiguity quantified in \cref{sec:layer0privacy}.

\subsection{Wrapper logic}

At a high level, the wrapper circuit performs four tasks.
\begin{enumerate}[leftmargin=1.9em]
    \item It detects dummy leaves by checking whether the child \texttt{block\_hash} is all zeros.
    \item It enforces that every non-dummy child agrees with the reference child's \texttt{block\_hash}, \texttt{asset\_id}, and \texttt{volume\_fee\_bps}.
    \item It computes grouped exit slots by summing amounts for identical exit accounts and zeroing duplicate slots.
    \item It forwards real nullifiers and substitutes random dummy nullifiers for dummy children.
\end{enumerate}

\subsection{Higher-layer aggregation}

Higher recursive layers operate over already aggregated proofs rather than raw leaf witnesses. This separation is important: the privacy-critical shuffle and dummy-handling logic stays local to the first layer, while later layers can be delegated because they receive opaque layer-0 aggregates rather than the client's raw proof set. In particular, a layer-1 aggregator learns only the public output of layer-0 and therefore does not observe the client's internal real-versus-dummy placement or pre-grouped exit set.

\subsection{Authenticated deposit records}

To tighten the security argument, it is useful to name the object that the proof is actually certifying.

\begin{definition}[Authenticated deposit record]
An \emph{authenticated deposit record} is a tuple
\[
d = (a, v_{\mathrm{in}}, c, \alpha_{\mathrm{fund}}, \alpha_{\mathrm{wh}}, h_B, b)
\]
for which there exists witness data \(w\) such that \((x,w) \in \mathcal{R}_{\mathrm{leaf}}\), where \(x\) exposes asset identifier \(a\), block commitment \(h_B\), block number \(b\), and exits satisfying the fee constraint against \(v_{\mathrm{in}}\). Intuitively, \(d\) is a deposit object that has been authenticated by the storage-proof and header-binding subcircuits described in \cref{sec:formal,sec:leaf}.
\end{definition}

A \emph{real} leaf is a leaf whose public \texttt{block\_hash} is nonzero. A \emph{dummy} leaf is a padding object identified by the sentinel \texttt{block\_hash} \(=0^{32}\) and used only inside first-layer aggregation.

\subsection{Completeness}

\begin{theorem}[Leaf completeness]
If an honest prover holds witness data \(w\) for which \((x,w)\in \mathcal{R}_{\mathrm{leaf}}\), then the prover can generate a leaf proof accepted by the leaf verifier.
\end{theorem}

\begin{proof}[Proof sketch]
Completeness follows from the correctness of the individual circuit fragments and of the proving system. Each fragment enforces a deterministic relation between its witness variables and public outputs. When the nullifier, wormhole address, storage proof, header commitment, fee inequality, and range constraints are all satisfied, the assembled witness satisfies \(\mathcal{R}_{\mathrm{leaf}}\), so the prover can produce an accepting proof.
\end{proof}

\begin{theorem}[Wrapper completeness]
Suppose a first-layer aggregator is given a full batch of valid child proofs, padding any missing positions with the designated dummy template. If all real children share the same public \texttt{block\_hash}, \texttt{asset\_id}, and \texttt{volume\_fee\_bps}, then the aggregator can produce an accepting first-layer wrapper proof.
\end{theorem}

\begin{proof}[Proof sketch]
The wrapper verifies the child proofs recursively, checks equality of the shared batch parameters across all non-dummy children, applies the implemented exit-grouping logic, and forwards nullifiers together with dummy replacements. When the inputs satisfy those conditions, the wrapper constraints are satisfiable by construction.
\end{proof}

\subsection{Leaf soundness and deposit binding}

\begin{theorem}[Leaf soundness]
Assume the leaf proof system is knowledge sound. For every accepted leaf proof \((x,\pi)\), there exists an extractor that outputs a witness \(w\) such that \((x,w)\in \mathcal{R}_{\mathrm{leaf}}\).
\end{theorem}

\begin{proof}[Proof sketch]
This is the standard consequence of knowledge soundness for the leaf relation. The extractor outputs values for the secret, counter, funding account, wormhole account, storage-proof witness, header fields, and input amount that satisfy conditions (C1)--(C6) from \cref{sec:formal}. No further claim is made here about whether the committed block actually exists in canonical chain history; that is deferred to the runtime check on the public \texttt{block\_hash}/\texttt{block\_number} pair.
\end{proof}

\begin{corollary}[Authenticated-deposit binding]
Assume leaf soundness and correct runtime validation of the public block commitment. Then every accepted real leaf proof binds to an authenticated deposit record and certifies that its disclosed exits satisfy the protocol fee rule for that record.
\end{corollary}

\begin{proof}
By leaf soundness, an accepting real leaf yields a witness satisfying \(\mathcal{R}_{\mathrm{leaf}}\). Conditions (C3) and (C4) bind the proof to an authenticated transfer record under the state root committed by the structured header fields, while condition (C5) enforces the fee inequality. Runtime validation upgrades the public header commitment from a merely self-consistent header witness to a real block in chain history.
\end{proof}

\subsection{First-layer wrapper soundness}

\begin{theorem}[First-layer wrapper soundness]
Assume the recursive verifier gadgets and the wrapper circuit are sound. Let \(Y\) be the public output of an accepted first-layer wrapper proof of arity \(N\). Then there exists a list of child instances
\[
((x_1,\pi_1,\delta_1),\ldots,(x_N,\pi_N,\delta_N)),
\]
where \(\delta_i\in\{0,1\}\) indicates whether child \(i\) is dummy, such that:
\begin{enumerate}[label=(W\arabic*),leftmargin=1.9em]
    \item if \(\delta_i=0\), then \(\pi_i\) is an accepted leaf proof on public input \(x_i\);
    \item if \(\delta_i=1\), then child \(i\) is recognized by the dummy sentinel \texttt{block\_hash} \(=0^{32}\);
    \item all non-dummy children share the same public header tuple exposed by \(Y\): a common block hash, asset identifier, fee parameter, and block number;
    \item the exit slots exposed by \(Y\) are exactly the result of applying the wrapper's grouping rule to the multiset of non-dummy child exits; and
    \item the nullifier region of \(Y\) contains every real child nullifier, while dummy positions are filled by wrapper-supplied replacement values.
\end{enumerate}
\end{theorem}

\begin{proof}[Proof sketch]
Soundness of the recursive verifier gadgets ensures that every accepted child proof referenced by the wrapper witness is itself valid. The wrapper then recomputes its public outputs from those children subject to explicit conditional checks: non-dummy children must agree on the batch \texttt{block\_hash}, \texttt{asset\_id}, and \texttt{volume\_fee\_bps}; exit slots are formed by the implemented grouping-and-zeroing logic; and nullifiers are forwarded child-by-child with dummy substitution in positions marked by the sentinel. The shared \texttt{block\_number} claim is obtained by combining wrapper soundness with leaf soundness: each real child is itself a sound leaf proof satisfying condition (C4), so a common \texttt{block\_hash} implies a common committed header and hence a common \texttt{block\_number}. A line-by-line version of this argument, together with the exact code-path linkage for dummy padding, is given in \cref{app:proofs}. Hence any accepting wrapper proof can be explained by a concrete list of real and dummy children satisfying (W1)--(W5).
\end{proof}

\begin{lemma}[Dummy-padding safety]
Assume the designated dummy template exposes zero-valued exits, and the runtime only settles the public exits produced by the wrapper. Then dummy padding cannot increase settled value. Random replacement of dummy nullifiers affects only accidental-collision probability and therefore liveness, not soundness.
\end{lemma}

\begin{proof}
Under the stated dummy-template assumption, a dummy child contributes no positive exit amount to the grouped wrapper outputs. Replacing its nullifier with a fresh filler does not create an authenticated deposit record or any additional settled value; at worst, an accidental nullifier collision causes rejection of the aggregate. Thus dummy padding cannot authorize value that was absent from the real children.
\end{proof}

\subsection{One-time withdrawal and double-spend resistance}

\begin{theorem}[One-time withdrawal per authenticated deposit record]
Assume:
\begin{enumerate}[label=(D\arabic*),leftmargin=1.9em]
    \item the runtime rejects reused nullifiers, including duplicates within one candidate aggregate submission;
    \item the authenticated transfer record fixes the transfer counter \(c\) used in nullifier derivation; and
    \item the hash function used in \(\WA(\cdot)\) and \(\Null(\cdot,\cdot)\) is collision resistant and second-preimage resistant for the relevant domain-separated inputs.
\end{enumerate}
Then no probabilistic polynomial-time adversary can obtain two accepted real withdrawals from the same authenticated deposit record except with negligible probability.
\end{theorem}

\begin{proof}[Proof sketch]
Suppose two accepted real withdrawals correspond to the same authenticated deposit record. By authenticated-deposit binding, extraction yields witnesses \((s,c)\) and \((s',c')\) together with the same authenticated record
\[
(a,v_{\mathrm{in}},c,\alpha_{\mathrm{fund}},\alpha_{\mathrm{wh}},h_B,b).
\]
Because the record fixes the counter, we have \(c=c'\). Because the record also fixes the wormhole address, both witnesses satisfy
\[
\WA(s)=\alpha_{\mathrm{wh}}=\WA(s').
\]
If \(s\neq s'\), the adversary has found a collision or second preimage for the wormhole-address derivation. Therefore \(s=s'\) except with negligible probability. But then
\[
\Null(s,c)=\Null(s',c'),
\]
so both withdrawals expose the same nullifier. The runtime rejects the second occurrence by assumption (D1). Hence one authenticated deposit record can underwrite at most one accepted real withdrawal.
\end{proof}

The theorem above is intentionally phrased in terms of an \emph{authenticated deposit record}, not an informal notion of ``the same funds.'' If a future protocol version supports explicit partial-withdrawal state transitions, that stronger state model should be proved separately rather than inferred from the present one-shot nullifier discipline.

\subsection{Operational unspendability and spend-path exclusivity}

The protocol needs one assumption that is partly external to the circuit: funds sent to an wormhole address should not also be directly spendable through an ordinary account-authentication path.

\begin{proposition}[Spend-path exclusivity under the chain's address discipline]
Let \(\mathsf{Addr}:\pkset\rightarrow\{0,1\}^{256}\) denote the chain's ordinary address derivation from externally controlled public keys. Suppose:
\begin{enumerate}[label=(U\arabic*),leftmargin=1.9em]
    \item every direct spend accepted by the base chain must be authorized by a valid public key \(pk\in\pkset\) whose address is \(\mathsf{Addr}(pk)\); and
    \item no efficient adversary can find \(pk\in\pkset\) such that \(\mathsf{Addr}(pk)=\WA(s)\) for a target wormhole address, except by breaking the collision or preimage resistance of the derivation.
\end{enumerate}
Then value sent to \(\WA(s)\) can leave the system only through the proof-authorized settlement path.
\end{proposition}

\begin{proof}[Proof sketch]
Any direct spend must use some valid external credential \(pk\) by assumption (U1). If such a spend exists from \(\WA(s)\), then \(\mathsf{Addr}(pk)=\WA(s)\), contradicting (U2) except with negligible probability. Therefore the only remaining spend path is the proof-authorized one. We treat spend-path exclusivity as a system assumption about the surrounding chain interface rather than as a theorem of the circuit in isolation.
\end{proof}

\subsection{Value conservation}

\begin{lemma}[Per-leaf fee bound]
For every accepted real leaf proof, the sum of its two public output amounts is at most the authenticated input amount net of the public fee parameter.
\end{lemma}

\begin{proof}
This is exactly condition (C5) of \(\mathcal{R}_{\mathrm{leaf}}\), obtained by extraction from leaf soundness.
\end{proof}

\begin{lemma}[Wrapper amount preservation]
Assume the first-layer wrapper groups identical exit accounts by summing their amounts into one slot and zeroing duplicate slots. Then the sum of the forwarded exit amounts equals the sum of the exit amounts of the real child proofs.
\end{lemma}

\begin{proof}
The wrapper's grouping step is a partition of the multiset of real child exits by destination account. Replacing each partition class by a single slot whose amount is the class sum preserves the total amount. Zeroing duplicate slots removes representation duplicates, not value.
\end{proof}

\begin{theorem}[Global value conservation]
Assume:
\begin{enumerate}[label=(V\arabic*),leftmargin=1.9em]
    \item leaf soundness,
    \item first-layer wrapper soundness,
    \item runtime rejection of reused nullifiers, and
    \item runtime settlement that applies exactly the forwarded public exit amounts.
\end{enumerate}
Then the total value settled by any accepted first-layer aggregate proof is at most the total value of its underlying authenticated deposit records after protocol fees. Across multiple accepted proofs, no authenticated deposit record is counted more than once.
\end{theorem}

\begin{proof}[Proof sketch]
By first-layer wrapper soundness, an accepted aggregate expands to a multiset of real child leaves and dummy padding. By dummy-padding safety, dummies contribute no settled value. By authenticated-deposit binding and the per-leaf fee bound, each real child contributes at most its authenticated deposit amount net of fees. By wrapper amount preservation, the sum of forwarded exit amounts equals the sum of the real child exit amounts. Finally, one-time withdrawal per authenticated deposit record rules out counting the same authenticated record twice via multiple accepted nullifiers. Hence settled value never exceeds the aggregate authenticated input value net of fees.
\end{proof}


\subsection{Layer-0 explanation classes and balance-linking difficulty}
\label{sec:layer0privacy}

The first aggregation layer does not hide the final exits or their amounts, but it does make leaf-level linkage more ambiguous. The key point is that the layer-0 wrapper is many-to-one: multiple ordered batches of real and dummy leaves can induce the same public aggregate once dummy nullifiers are replaced and duplicate exits are grouped.

\begin{definition}[Layer-0 explanation class]
Fix a first-layer public output \(Y\). The \emph{explanation class} \(\mathcal{E}(Y)\) is the set of ordered batches of real leaves and dummy leaves, together with per-leaf two-output assignments, that are consistent with the leaf fee rule and that produce \(Y\) under the implemented layer-0 padding, shuffle, grouping, and dummy-nullifier-substitution logic.
\end{definition}

A balance-linking adversary who sees only \(Y\) must distinguish among the elements of \(\mathcal{E}(Y)\). By contrast, a delegated layer-0 aggregator that receives the client's raw leaf proofs before padding and shuffling learns one concrete preimage of \(Y\) and therefore collapses this ambiguity. This is the formal reason that layer-0 is privacy-sensitive and should be client-local.

\begin{proposition}[Client-local layer-0 dominates delegated layer-0 for linkage privacy]
Fix any first-layer public output \(Y\). A downstream observer when layer-0 is performed client-locally receives only \(Y\), whereas a delegated layer-0 aggregator receives both \(Y\) and one specific preimage \(B \in \mathcal{E}(Y)\). Under a uniform prior over \(\mathcal{E}(Y)\), the optimal exact leaf-to-output linkage success probability is \(1/|\mathcal{E}(Y)|\) for the former view and \(1\) for the latter.
\end{proposition}

\begin{proof}
The downstream observer receives only \(Y\), so any exact linkage guess is a guess of which element of \(\mathcal{E}(Y)\) generated the aggregate. Under the stated prior, the optimal strategy succeeds with probability \(1/|\mathcal{E}(Y)|\). A delegated layer-0 aggregator, however, observes the raw child proofs before padding and shuffling, so it knows the realized preimage \(B\) directly.
\end{proof}

The size of \(\mathcal{E}(Y)\) grows combinatorially whenever multiple hidden child outputs collapse into one grouped exit.

\begin{proposition}[Combinatorial lower bound from grouped exits]
Let \(\Delta = 0.01\) be the circuit quantization unit. Fix a grouped exit account \(a\) in \(Y\) whose public forwarded amount is \(S_a \in \Delta \mathbb{Z}_{>0}\). If that total arose from \(r_a \ge 1\) positive real child outputs to the same account, then the number of positive quantized decompositions of \(S_a\) into \(r_a\) hidden child contributions is
\[
\binom{S_a/\Delta - 1}{r_a - 1}.
\]
Consequently, if \(A^+\) denotes the set of grouped exit accounts that received contributions from more than one real child output, then
\[
|\mathcal{E}(Y)| \ge \prod_{a \in A^+} \binom{S_a/\Delta - 1}{r_a - 1},
\]
with additional multiplicative ambiguity coming from hidden dummy placement, private batch order, and the assignment of at most two outputs to each real leaf.
\end{proposition}

\begin{proof}
For a fixed account \(a\), write \(S_a = m_a \Delta\) with \(m_a \in \mathbb{Z}_{>0}\). Counting positive integer solutions to
\[
x_1 + \cdots + x_{r_a} = m_a
\]
gives the classical stars-and-bars count \(\binom{m_a - 1}{r_a - 1}\). Each such solution corresponds to a distinct quantized decomposition of the grouped amount into hidden child outputs. Independent grouped accounts multiply, yielding the stated lower bound. Any uncertainty about which leaves were dummy, how the client ordered the batch before wrapping, or which of a leaf's two output positions carried a given contribution can only enlarge the explanation class.
\end{proof}

The implementation preserves one real child in slot~0. This reduces but does not eliminate occupancy ambiguity from shuffling.

\subsection{Privacy boundary}

Two privacy aspects:

\emph{Proof-level witness hiding}: Secrets, funding accounts, storage paths, and header details are not public inputs. Leakage would require breaking the proof system or side channels.

\emph{System-level privacy}: Output amounts, exit accounts, nullifiers, and block references are public. Layer-0 padding enlarges the explanation class, but this is not full unlinkability---amount leakage, address reuse, and timing correlation remain limitations.

\subsection{Methodology}

Benchmarks were executed on an Apple M2 Max with 12 CPU cores and 32~GB of RAM, running macOS 26.3.1 with \texttt{rustc 1.90.0} and \texttt{cargo 1.90.0}. To reduce interference from prior builds, runs used scratch-local \texttt{HOME}, \texttt{CARGO\_HOME}, \texttt{TMPDIR}, and \texttt{CARGO\_TARGET\_DIR} rooted under \texttt{/tmp}.

For the implemented first aggregation layer, we evaluated same-\texttt{block\_hash} batches with \(N \in \{2,4,8,9,16,25,32,36,49\}\) leaf proofs. For the next recursive layer, we fixed each layer-0 proof to aggregate eight leaves and varied the number of input layer-0 proofs \(M \in \{2,4,8,16,32\}\).

All first-layer measurements use repeated identical dummy leaf proofs, so they remain within the implemented same-block / shared-\texttt{block\_hash} protocol surface; they do not measure heterogeneous real proofs drawn from a live block. The second-layer runs likewise use repeated dummy-derived layer-0 proofs with full batches only.

Reported build times are builder wall-clock times. For layer~0, build includes leaf and layer-0 artifact generation plus configuration hashing. For layer~1, build measures layer-1 artifact generation only, with prerequisite layer-0 artifacts generated separately. Reported prove times include backend load/deserialization and the \texttt{commit()} stage. \section{Limitations}
\label{sec:discussion}

\paragraph{Same-block scoping.}
Layer-0 batches proofs sharing the same \texttt{block\_hash}. Cross-block aggregation is future work.

\paragraph{Two-output limit.}
Each leaf supports two exits (spend + change), not general multi-recipient batching.

\paragraph{Amount quantization.}
Amounts are quantized to 0.01-unit resolution to reduce circuit cost.

\paragraph{Privacy leakage.}
Output amounts, exit addresses, and timing are public. Layer-0 padding enlarges the explanation class (\cref{sec:layer0privacy}), but this is plausible deniability, not full shielding.

\section{Related Work}
\label{sec:related}

\paragraph{Wormholes and privacy pools.}
EIP-7503 proposes proof-of-burn reminting via secret-derived addresses \citep{eip7503}. Privacy Pools adds compliance-aware privacy sets \citep{privacypools}. Quantus implements recursive proofs over authenticated chain state.

\paragraph{Shielded systems.}
Zcash provides strong privacy through shielded notes; Tornado Cash popularized mixer-style privacy on Ethereum \citep{zcash,tornadocash}. Quantus offers lighter-weight private settlement anchored in native chain state.

\paragraph{Recursive proofs.}
Halo and Plonky2 are standard recursive proof systems \citep{halo,plonky2}. Plonky3 provides Poseidon2 primitives with a separate recursion track \citep{plonky3,plonky3recursion}. Quantus applies recursive composition to post-quantum private settlement using \texttt{qp-plonky2} with Poseidon2.

\paragraph{Post-quantum blockchains.}
ML-DSA adoption raises questions about transaction size and verification cost \citep{mldsa}. Quantus compresses authorization into ZK proofs rather than publishing large signatures on-chain.

\section{Conclusion}
\label{sec:conclusion}

Wormhole addresses let a post-quantum blockchain move authorization off-chain while preserving authenticated settlement. Users deposit to secret-derived addresses, prove the deposit exists and hasn't been spent, and reveal only settlement outputs. Recursive aggregation amortizes verification. We formalized the leaf relation, analyzed layer-0 privacy, proved security properties, and reported benchmark timings.

\subsection{Deposit binding}

For a real leaf proof (non-zero \texttt{block\_hash}), knowledge soundness extracts a witness satisfying \(\mathcal{R}_{\mathrm{leaf}}\): (C3) gives an authenticated storage proof, (C4) binds the header commitment, and the runtime checks the block exists in canonical history. This is the only step outside the circuit---the runtime validates the \((\texttt{block\_hash}, \texttt{block\_number})\) pair.

\subsection{Wrapper soundness}

The wrapper verifies \(N\) child proofs and enforces: non-dummy children share \texttt{block\_hash}, all children share \texttt{asset\_id} and \texttt{volume\_fee\_bps} (from slot~0). Exit grouping sums amounts by destination and zeros duplicates. Nullifiers forward real values or dummy fillers based on the sentinel check.

Common \texttt{block\_number} follows from leaf soundness: real children with the same \texttt{block\_hash} have the same committed header, hence the same \texttt{block\_number}.

\subsection{Dummy-padding safety}

Dummies have zero outputs and zero exit accounts, so they contribute nothing to forwarded amounts. The wrapper uses fresh random fillers for dummy nullifiers---this affects liveness (collision risk) not soundness.

\subsection{Layer-0 ambiguity}

Three sources enlarge the explanation class \(\mathcal{E}(Y)\): (1) exit grouping hides individual contributions, (2) dummy nullifiers look random, (3) shuffle hides slot assignment. For grouped exits with total \(S_a = m_a \Delta\) and multiplicity \(r_a\), there are \(\binom{m_a - 1}{r_a - 1}\) compositions---the stars-and-bars count from the main text.

\subsection{One-time withdrawal}

Two withdrawals from the same deposit record \(d\) would extract witnesses with the same counter \(c\) and wormhole address, hence \(\WA(s)=\WA(s')\). Unless \(s=s'\) (collision), both expose the same nullifier \(\Null(s,c)\). The runtime rejects duplicates.

\subsection{Value conservation}

By wrapper soundness, aggregates expand into real children and dummies. Dummies contribute nothing. Each real child is bound to an authenticated deposit with a fee-constrained exit sum. Nullifier freshness prevents double-counting. Total settled value is bounded by total deposit value.

\section{Privacy-score heuristic}

A wallet-side heuristic (not a formal guarantee): for candidate deposits \(\mathcal{D}\) and withdrawal amount \(V\),
\[
\mathsf{Score}(V,\mathcal{D}) = \log_2 \left| \left\{ S \subseteq \mathcal{D} : \sum_{d\in S} d\cdot (1-f) \approx V \right\} \right|.
\]
